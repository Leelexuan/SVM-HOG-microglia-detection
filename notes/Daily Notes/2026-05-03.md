### Sliding Window Detection — HOG is Already Efficient

HOG features are correctly precomputed for the **full image** once using `feature_vector=False`, then sliced per window. This was already correct before any optimisation. The actual per-window bottlenecks were:

- **LBP** (most expensive): `local_binary_pattern` called on every 64×64 crop

- **Intensity stats**: iterating over 4096 pixels per window per channel

- **Scaler/PCA/classifier**: called individually once per window

  

### Sliding Window Optimisations Applied

`sliding_window_detect` in `cell-detection` was refactored to precompute where possible:

  

| Feature | Before | After |

|---|---|---|

| HOG coarse/fine | Already precomputed (full-image map + slice) | Unchanged |

| LBP | `local_binary_pattern` on 64×64 per window | Computed once on full image, sliced per window |

| Intensity mean/std/skew | Loop over 4096 pixels per window | `cv2.integral` summed-area tables — O(1) per window |

| Scaler/PCA/clf | 1 call per window | Batched every 512 windows |

| Spatial bins | Per window (requires resize) | Cannot precompute — unchanged |

| Color histogram | Per window (adaptive bin range) | Cannot precompute — unchanged |

  

**Skewness formula via integral images:**

`skew = (E[x³] - 3μE[x²] + 2μ³) / σ³`

Uses three integral images: sum, sum-of-squares, sum-of-cubes per channel.

  

**LBP note:** Precomputed LBP map uses full-image neighbourhood for border pixels, which is slightly more accurate than per-patch computation at training time (minor, acceptable discrepancy).

  

### Why the SVM-as-Convolution Trick Doesn't Apply Here

A LinearSVC weight vector can be treated as a convolution filter over the HOG map, avoiding the sliding window loop entirely. This only works when:

1. Features are HOG-only (no color, no LBP, no spatial bins)

2. No PCA (PCA mixes all features, destroying the HOG spatial structure)

  

This notebook uses a multi-feature vector and PCA, so the trick is not applicable. The precomputed HOG maps + slicing approach is the correct middle ground.

  

### Critical Issue: Patch Metrics ≠ Detection Quality

**The most important finding from this session.**

  

The training/evaluation pipeline measures patch classification F1 (~0.92). The detection pipeline performance on full images is much worse. These measure different things:

  

- **Patch evaluation**: balanced split, ~50% positive patches, classification accuracy

- **Detection inference**: ~15,000 windows per scale on a 2048×2048 image, with only 10–20 actual cells — extreme class imbalance not seen during training

  

A 92% accurate patch classifier still produces hundreds of false positives when sweeping a full image.

  

**The detection threshold was tuned on the wrong signal.** `find_optimal_threshold` sweeps thresholds to maximise patch-level F1. The correct threshold for detection should maximise IoU-matched bounding box precision/recall on full images, and will be substantially higher (more conservative).

  

### Recommended Fixes (Priority Order)

  

1. **Tune threshold on detection output**: sweep `cfg.detection_threshold` and measure bounding box precision/recall (IoU ≥ 0.5) on training/validation images directly, not patch F1.

  

2. **Hard negative mining** (the `Hard Negative Mining/` folder is for this): run the detection pipeline on training images, collect every false positive window crop, add to the noise training set, retrain. This directly teaches the classifier about the specific background regions it confuses at inference time.

  

3. **Make detection-level precision/recall the primary validation metric**: patch accuracy is useful for debugging feature quality but is not a reliable proxy for detection quality given the extreme inference-time class imbalance.
