## Goal — CV Interview Question Bank

Structured set of CV interview questions and answers, organised by topic. Mix
of difficulty levels. For each question the answer gives the core concept plus
the nuance an interviewer pushes on. Not project-specific — broad coverage to
prep for the unpredictable parts of the interview.

---

## 1. Classical CV fundamentals

**Q: What's the difference between convolution and cross-correlation?**
A: Convolution flips the kernel before applying it; cross-correlation doesn't.
In CNNs we call the operation "convolution" but it's actually cross-correlation
— the flip doesn't matter since the kernel is learned. Mathematically, true
convolution is commutative; cross-correlation isn't.

**Q: What does a Gaussian kernel do, and why is it used everywhere?**
A: It performs weighted local averaging, with weights falling off by distance
from the centre. Used because (1) it suppresses high-frequency noise while
preserving structure better than a box filter, (2) it's separable — a 2D
Gaussian = 1D Gaussian × 1D Gaussian, so an N×N filter becomes 2N operations
not N², (3) it's the unique kernel that doesn't introduce new extrema (no
ringing artifacts).

**Q: How does Canny edge detection work?**
A: Five steps: (1) Gaussian smoothing, (2) gradient computation (Sobel), (3)
non-max suppression along the gradient direction (thins edges to 1px), (4)
double thresholding (strong/weak edges), (5) hysteresis — connect weak edges
to strong neighbours. The hysteresis step is what makes Canny robust vs naïve
gradient thresholding.

**Q: What are HOG features?**
A: Divide the image into small cells (e.g., 8×8 px), compute the gradient at
each pixel, then build a histogram of gradient orientations (typically 9 bins
covering 0-180°) per cell, weighted by gradient magnitude. Group cells into
overlapping blocks and L2-normalise each block to gain illumination invariance.
Concatenate everything into a feature vector. Captures local edge structure
while being translation-tolerant and illumination-invariant.

**Q: HOG vs SIFT — when would you use which?**
A: HOG is dense (computed everywhere on a grid) — designed for *detection* (is
there an object in this window?). SIFT is sparse (computed at keypoints) and
includes rotation/scale invariance — designed for *matching* (do these two
images contain the same object?). HOG is faster but not rotation-invariant;
SIFT is slower but matches across viewpoints.

**Q: What is Local Binary Pattern (LBP)?**
A: For each pixel, compare its value to its 8 neighbours, encode each
comparison as a bit (1 if neighbour is brighter), concatenate into an 8-bit
pattern. Then histogram these patterns over a region. Captures local *texture*
rather than edges. "Uniform" LBP collapses patterns by number of bit-transitions,
giving rotation invariance.

**Q: Why are integral images useful?**
A: They let you compute the sum of any rectangular region in O(1) by 3
additions/subtractions: `S(a,b,c,d) = II(d) − II(b) − II(c) + II(a)`. Critical
for Viola-Jones-style Haar features at scale — without integral images,
computing thousands of Haar features per window would be intractable.

## 2. Object detection

**Q: Explain sliding window detection.**
A: Slide a fixed-size window across an image at some stride, classify the
window contents at each position, accumulate detection scores. Handle scale
via an image pyramid (resize the image, repeat). Final step: NMS to collapse
overlapping detections. Foundational for classical detection (Viola-Jones,
Dalal-Triggs) and conceptually present in modern anchor-based detectors.

**Q: What is non-maximum suppression and why do we need it?**
A: Detectors fire multiple overlapping boxes around each object. NMS collapses
them to one. Greedy NMS: sort by score, pick top, suppress everything with
IoU > threshold (e.g., 0.5), repeat on survivors. Without NMS, mAP collapses
because every true object gets 5-20 nearly-identical predictions.

**Q: What's Soft-NMS and why was it proposed?**
A: Greedy NMS silently kills *true positives* in crowded scenes — if two
objects overlap with IoU > threshold, NMS suppresses the lower-scoring one
even if it's a real object. Soft-NMS (Bodla 2017) decays scores by a continuous
function of IoU instead of hard suppression, so overlapping true positives
survive with reduced confidence rather than disappearing.

**Q: What's IoU and how do you compute it?**
A: Intersection over Union: area of overlap divided by area of union of two
boxes. For axis-aligned boxes:
`IoU = max(0, min(x2a,x2b) - max(x1a,x1b)) * max(0, min(y2a,y2b) - max(y1a,y1b)) / (area_a + area_b - intersection)`.
Used for matching predictions to GT and for NMS.

**Q: Two-stage vs one-stage detectors — what's the trade-off?**
A: Two-stage (Faster R-CNN): RPN proposes regions → classifier refines. Higher
accuracy, slower. One-stage (YOLO, SSD, RetinaNet): direct dense prediction.
Faster, historically lower mAP. The accuracy gap closed with focal loss
(RetinaNet) and architectural improvements (YOLOv4+). Modern recommendation:
one-stage for real-time, two-stage when accuracy is critical and latency
permits.

**Q: Walk through R-CNN → Fast R-CNN → Faster R-CNN.**
A:
- **R-CNN** (2014): selective search proposes ~2k regions, crop each, run
  through CNN, classify. Very slow (~50s/image) because the CNN runs 2k times.
- **Fast R-CNN** (2015): run CNN *once* on the whole image, then RoI-Pool
  features from the shared feature map. ~10× faster.
- **Faster R-CNN** (2015): replace selective search with a learned Region
  Proposal Network (RPN) sharing the CNN backbone. Region proposals become
  learned and fast. ~5-10 FPS.

**Q: What's the difference between RoI Pooling and RoI Align?**
A: RoI Pooling quantises both the RoI boundaries and the bin boundaries to
integer feature-map coordinates — introduces misalignment that's fine for
classification but breaks mask prediction. RoI Align uses bilinear
interpolation at exact sub-pixel locations, no quantisation. Mask R-CNN's main
practical contribution.

**Q: What is an anchor box?**
A: A predefined box of a particular size/aspect ratio, placed at every position
on a feature map. The detector predicts offsets to anchors rather than absolute
box coordinates — much easier to learn (small deltas vs raw pixel coordinates).
Typically 3-9 anchors per location, covering different scales and aspect
ratios. Modern anchor-free methods (FCOS, CenterNet) bypass these.

**Q: What is DETR and why is it different?**
A: DEtection TRansformer (Carion 2020) — frames detection as a set-prediction
problem with a transformer encoder-decoder. Eliminates anchors and NMS
entirely; uses bipartite matching loss (Hungarian algorithm) to assign
predictions to GT. End-to-end trainable, no hand-crafted post-processing. Slow
to train and high latency; spawned faster variants (Deformable DETR, DINO).

**Q: How does YOLO work in one sentence?**
A: Divide the image into a grid; each grid cell predicts B bounding boxes +
class probabilities directly from a single CNN forward pass; apply NMS. "You
only look once" = no separate proposal stage.

## 3. Segmentation

**Q: Semantic vs instance vs panoptic segmentation?**
A: Semantic: pixel-level class labels, no distinction between object instances
("all car pixels"). Instance: pixel masks for each individual object instance
("car 1, car 2"). Panoptic: union of both — every pixel gets a (class, instance
ID) tuple. Stuff classes (sky, road) get class only; thing classes (car,
person) get class + instance.

**Q: Explain the U-Net architecture.**
A: Symmetric encoder-decoder. Encoder downsamples (conv + pool), capturing
semantic content. Decoder upsamples (transposed conv or upsample + conv),
recovering spatial resolution. **Skip connections** copy encoder features at
each resolution to the matching decoder layer — preserves fine spatial detail
that gets lost in downsampling. Designed for biomedical imaging with small
datasets; works extremely well there.

**Q: Why are skip connections critical in segmentation?**
A: Downsampling loses spatial precision. The decoder can recover the *what*
(semantic class) from low-resolution features but not the *where* (exact
boundary). Skip connections re-inject high-resolution features so the boundary
prediction is sharp. Without them, segmentation masks are blobby.

**Q: How does Mask R-CNN add segmentation to Faster R-CNN?**
A: Adds a parallel mask-prediction branch alongside the existing classification
+ box-regression branches. The mask branch is a small FCN operating on each
RoI-aligned feature, predicting a binary mask per class. The keys are (1) RoI
Align (not RoI Pool) for pixel-accurate alignment, (2) per-class mask
prediction (avoids inter-class competition).

**Q: What is SAM (Segment Anything Model)?**
A: A foundation model for promptable segmentation (Meta, 2023). Given a prompt
(point, box, mask, or text), outputs a segmentation mask. Trained on 1.1B masks
from 11M images. Zero-shot generalises to new domains. Practical use: get masks
fast without training a segmenter; useful for medical imaging as a labelling
accelerator.

## 4. Convolutional architectures

**Q: What is receptive field?**
A: The region of the input image that influences a single output activation.
Grows with depth (each conv layer expands it). Pool/stride operations expand
it faster. Critical concept: if the receptive field is smaller than the object
you're trying to detect, the network can't see it whole.

**Q: Why use 1×1 convolutions?**
A: Three uses: (1) **channel-wise mixing** — linearly combines features across
channels at each spatial position, no spatial mixing; (2) **dimension reduction**
— cheap way to drop from 512 channels to 128 before an expensive 3×3 conv (the
"bottleneck" pattern in ResNet/Inception); (3) **non-linearity insertion** —
followed by an activation, adds depth without spatial change.

**Q: Why do residual connections help?**
A: Two reasons: (1) **gradient flow** — they provide a direct path for
gradients to flow backwards, mitigating vanishing gradients in deep networks;
(2) **identity mapping prior** — the network only needs to learn the *residual*
(deviation from identity), which is often closer to zero than learning the full
mapping from scratch. Result: networks can go from ~22 layers (VGG) to 1000+
(ResNet-1001) without degradation.

**Q: Dilated/atrous convolution — what's it for?**
A: A conv with gaps in the kernel — dilation rate r means the kernel samples
every r-th pixel. Expands receptive field exponentially without losing
resolution or adding parameters. Critical in semantic segmentation (DeepLab)
where you need a large receptive field but can't afford to downsample further.

**Q: Depthwise separable convolution?**
A: Decomposes a standard conv into (1) depthwise — one filter per input
channel, no cross-channel mixing, then (2) pointwise — 1×1 conv to mix channels.
Reduces parameters from O(k²·C_in·C_out) to O(k²·C_in + C_in·C_out). Foundation
of MobileNet — ~8-9× fewer parameters at similar accuracy.

**Q: Why does pooling exist if we have stride?**
A: Historically pooling was the way to downsample; strided convs are now often
preferred (learnable). Pooling adds local translation invariance and is
parameter-free, but discards information non-adaptively. Strided convs learn
what to keep. Modern architectures often use one or the other but rarely both
at the same downsample step.

**Q: What's the difference between BatchNorm, LayerNorm, GroupNorm, InstanceNorm?**
A: All normalise activations but compute mean/variance over different axes:
- **BatchNorm**: per-channel, over (batch, spatial) — depends on batch statistics
- **LayerNorm**: per-sample, over (channel, spatial) — independent of batch
- **InstanceNorm**: per-sample, per-channel, over spatial — useful for style transfer
- **GroupNorm**: per-sample, over (channel-group, spatial) — middle ground

Use LayerNorm for transformers (batch independence), GroupNorm when batch sizes
are small (e.g., 1-2 for medical 3D images), BatchNorm for vision with large
batches.

**Q: Why does BatchNorm help training?**
A: Original story: reduces "internal covariate shift" — keeping activation
distributions stable. Newer story (Santurkar 2018): it smooths the loss
landscape, making gradients more predictable, allowing higher learning rates.
Empirically: it acts as regulariser, accelerates training, reduces sensitivity
to initialization.

## 5. Loss functions

**Q: What is focal loss and what problem does it solve?**
A: Standard cross-entropy gives equal loss weight to all examples. With extreme
class imbalance (e.g., 1:1000 in dense detection), the loss is dominated by
easy negatives — gradients flow mostly from background patches the model
already gets right. Focal loss multiplies the loss by `(1 − p_t)^γ` where p_t
is the predicted probability of the true class. This down-weights easy examples
(high p_t) so gradients flow from hard examples instead. γ=2 is standard.

**Q: Compare IoU loss, GIoU, DIoU, CIoU.**
A:
- **IoU loss**: 1 − IoU. Zero gradient when boxes don't overlap.
- **GIoU**: adds a penalty for the smallest enclosing box around both, so
  non-overlapping boxes still get a gradient.
- **DIoU**: adds a normalised centre-distance term; converges faster.
- **CIoU**: DIoU + aspect-ratio consistency. Used in YOLOv4+.

**Q: Dice loss for segmentation — why?**
A: Cross-entropy in segmentation is dominated by the majority class (often
background). Dice loss = `1 − 2|A ∩ B| / (|A| + |B|)` directly optimises
overlap, naturally handling class imbalance. Combined loss `BCE + Dice` is the
practical default for medical segmentation.

**Q: When would you use triplet loss?**
A: Metric learning — when you want to learn an embedding where similar items
are close and dissimilar items are far. Used in face recognition, person re-ID,
image retrieval. Triplet = (anchor, positive, negative); minimise
`d(a,p) − d(a,n) + margin`. Hard-negative mining matters a lot — random
triplets are mostly trivial.

## 6. Training techniques

**Q: Why use learning rate warmup?**
A: Early in training, gradients are noisy and weights are random. A high LR
amplifies that noise and can destabilise training (especially with BatchNorm
where statistics are unreliable initially). Warmup: ramp LR from 0 to target
over the first N steps, then schedule normally. Standard for transformer
training; helps large-batch CNN training.

**Q: What is label smoothing?**
A: Instead of one-hot targets (e.g., [0,0,1,0]), use slightly smoothed labels
(e.g., [0.025, 0.025, 0.925, 0.025]). Prevents the model from becoming
over-confident, which improves calibration and sometimes accuracy. Equivalent
to adding a uniform prior. Common in modern image classification (~0.1
smoothing).

**Q: What does dropout actually do?**
A: Randomly zeros activations during training with probability p, scales
remaining by 1/(1−p). Two interpretations: (1) approximates an ensemble of
exponentially many sub-networks, (2) prevents co-adaptation — neurons can't
rely on specific other neurons being present, forcing distributed
representations. Less common in modern CV (BatchNorm + heavy augmentation
replaces it) but standard in transformers.

**Q: Gradient accumulation — when and why?**
A: When you want a larger effective batch size than GPU memory allows. Do N
forward+backward passes accumulating gradients, then step the optimiser once.
Equivalent to batch size = N × physical batch, modulo BatchNorm which only sees
the physical batch. Trade: slower wall-clock per parameter update.

**Q: Mixed precision training — what is it?**
A: Train with fp16 (or bf16) for forward/backward passes, keep an fp32 master
copy of weights for the optimizer update. ~2× faster and ~2× less memory on
modern GPUs. fp16 has limited dynamic range so you need loss scaling to prevent
gradient underflow; bf16 doesn't (wider exponent, narrower mantissa). PyTorch
AMP / TF mixed precision handle this automatically.

## 7. Evaluation metrics

**Q: When do you use precision-recall curve vs ROC curve?**
A: ROC plots TPR vs FPR. PR plots precision vs recall. **PR is preferred under
heavy class imbalance** — ROC can look misleadingly good when negatives vastly
outnumber positives, because FPR stays low even with many false positives. Rule
of thumb: medical detection / object detection → PR. Balanced classification →
either.

**Q: Explain mAP and COCO mAP@[0.5:0.95].**
A: For each class, compute Average Precision (area under the PR curve), then
average over classes. PASCAL VOC uses IoU=0.5 threshold for matching. COCO
averages AP over 10 IoU thresholds (0.5 to 0.95 in steps of 0.05) — harsher
because it rewards tight box fits. COCO mAP@0.5 is much closer to VOC mAP than
COCO mAP@[0.5:0.95].

**Q: What's the difference between F1, F2, and F0.5?**
A: Fβ = (1+β²) × P×R / (β²×P + R). β=1 weighs precision and recall equally.
β=2 emphasises recall (e.g., medical screening where misses are expensive).
β=0.5 emphasises precision (e.g., spam detection where false positives are
annoying).

**Q: Why isn't accuracy a good metric in detection?**
A: Detection has no "true negative" class for whole images — almost every pixel
is "background", so accuracy is trivially high. Precision/recall measure what
matters (did you find cells, and were your detections real cells?).

## 8. Data augmentation

**Q: What's the difference between Mixup and CutMix?**
A: Mixup linearly interpolates two images and their labels:
`x_mix = λx1 + (1−λ)x2`, label is `λy1 + (1−λ)y2`. CutMix cuts a rectangular
patch from image B and pastes it into image A; label is weighted by patch area.
CutMix preserves local image statistics (no blurry interpolations), works
better in practice for detection/localisation.

**Q: What is AutoAugment vs RandAugment?**
A: AutoAugment searches for the optimal augmentation policy using RL —
expensive (thousands of GPU hours). RandAugment simplifies: just two
hyperparameters (number of augmentations to apply, magnitude). Nearly as good,
far cheaper. The lesson: augmentation policy matters but it's robust — you
don't need to over-engineer it.

**Q: Test-time augmentation (TTA) — what is it?**
A: At inference, apply augmentations (flips, rotations, crops), predict on
each, ensemble the predictions. Improves accuracy by 0.5-2% typically, at N×
inference cost. Useful for competitions / final benchmarks; rarely worth it
in production.

**Q: What augmentations should you AVOID in medical imaging?**
A: Anything that distorts diagnostic features:
- Aggressive colour shifts (stain colour is informative)
- Vertical flips for body-orientation-aware tasks (e.g., chest X-rays — heart
  is on the left)
- Heavy elastic deformation if measurement matters
- Mixup when labels are clinically meaningful (mixed labels don't correspond
  to any real case)

Safe: rotations, small translations, brightness jitter within calibration
tolerance.

## 9. Transfer learning

**Q: When should you freeze layers vs fine-tune everything?**
A: Rule of thumb based on dataset size + similarity to pretraining:
- Small + similar (e.g., 1k images of cats from ImageNet-style natural images):
  freeze most, train classifier head
- Small + dissimilar (e.g., 1k microscopy images): freeze early layers
  (low-level features generalise), fine-tune later layers
- Large + dissimilar: fine-tune everything; pretrained init still helps but
  the data is enough to learn task-specific features
- Large + similar: fine-tune everything, lower LR than from-scratch

**Q: What is catastrophic forgetting in transfer learning?**
A: When fine-tuning destroys the pretrained features that were useful.
Mitigations: low LR for pretrained layers, gradual unfreezing, layer-wise LR
schedule, regularisation toward pretrained weights (EWC). Most acute when the
new task is small or when fine-tuning for many epochs.

**Q: Why does ImageNet pretraining help on medical images that look nothing
like ImageNet?**
A: Low-level features (edges, textures, corners) are universal — they transfer
regardless of domain. Higher layers are more domain-specific, but starting from
non-random init speeds convergence and improves final accuracy especially with
limited data. The benefit shrinks as the dataset grows; with millions of
medical images, training from scratch can match or exceed pretrained.

## 10. Self-supervised learning / pretraining

**Q: What is contrastive learning (SimCLR, MoCo)?**
A: Train an encoder to map two augmentations of the same image to similar
embeddings (positive pair), and different images to different embeddings
(negative pairs). Loss: InfoNCE — basically softmax over similarity scores.
SimCLR uses large in-batch negatives; MoCo maintains a queue of historical
embeddings to get many negatives without huge batches. Produces representations
that match supervised pretraining quality on many downstream tasks.

**Q: What is MAE (Masked Autoencoder)?**
A: Randomly mask 75% of image patches, train a transformer to reconstruct them
from the visible 25%. Heavy masking is the trick — makes the task hard enough
that the model learns semantic structure, not just local interpolation. Encoder
operates only on visible patches (fast); decoder is small and lightweight.
Strong pretraining for ViTs.

**Q: What is CLIP and why is it useful?**
A: Contrastive Language-Image Pretraining (OpenAI 2021). Trains image and text
encoders jointly: embeddings of an image and its caption should be close;
embeddings of mismatched pairs should be far. Trained on 400M image-text pairs
from the web. Useful for: (1) zero-shot classification (compare image embedding
to text-prompted class embeddings), (2) image search, (3) as a frozen feature
extractor for downstream tasks, (4) as a building block for multimodal models.

## 11. Vision Transformers

**Q: How does a ViT work?**
A: (1) Split image into N patches (e.g., 16×16), (2) linearly project each
patch to a vector (the "patch embedding") + add positional encoding, (3) feed
sequence to standard transformer encoder, (4) take the [CLS] token output for
classification. Replaces the inductive bias of CNNs (locality, translation
invariance) with capacity + data.

**Q: Why do ViTs need so much data compared to CNNs?**
A: CNNs have built-in priors (local connectivity, weight sharing across
positions, translation equivariance) that ViTs lack. With limited data, those
priors substitute for examples — a CNN "knows" that learning a cat detector at
position X means it should work at position Y. ViTs have to *learn* this from
data. So ViTs underperform CNNs at small scale (ImageNet-1k) but exceed them
at large scale (JFT-300M, LAION).

**Q: What's special about Swin Transformer?**
A: Computes attention within local windows (not globally) to make compute
linear in image size. The "shifted windows" trick: alternate layers shift the
window grid by half a window, so information flows across window boundaries.
Restores hierarchical feature pyramid structure (like CNNs), enabling drop-in
replacement in detection / segmentation pipelines.

## 12. Medical imaging / pharma-specific

**Q: What is DICOM and why does it matter?**
A: Digital Imaging and Communications in Medicine — the standard format for
medical imaging. Combines pixel data with extensive metadata (modality,
acquisition parameters, patient ID, study info). Practical implications:
(1) you can't just `cv2.imread` it — need pydicom, (2) metadata can leak
identity (HIPAA), (3) pixel values aren't always 8-bit — CT is 12-16-bit
Hounsfield units.

**Q: What is multiple instance learning and where does it apply?**
A: A weak-label setting: each *bag* (e.g., whole-slide histopathology image)
has a label, but individual *instances* (patches) do not. Standard MIL: a bag
is positive if at least one instance is positive. Used heavily in pathology
because annotating every cell is infeasible; slide-level diagnoses are
available.

**Q: How would you approach cell counting in microscopy?**
A: Three approaches in increasing complexity:
1. **Detection-then-count**: detect each cell with a bounding box, count
   detections. Robust, interpretable, requires per-cell labels.
2. **Density-map regression**: predict a density map that integrates to the
   count (U-Net regresses the map). Faster than detection, handles crowded
   scenes, doesn't give per-cell info.
3. **Direct regression**: predict count directly with a CNN regressor. Simplest,
   no localization, hardest to debug.

Choice depends on: do you need *which cells* (drug treatment effect per cell)
or just *how many* (assay readout).

**Q: Why is interpretability extra important in medical AI?**
A: (1) Regulatory: FDA/EMA require explainability for clinical decision
support. (2) Trust: clinicians won't act on a model they can't reason about.
(3) Failure modes: a wrong prediction in medical AI can harm a patient;
understanding *why* it failed is critical for safety. (4) Bias auditing:
medical AI famously fails on under-represented populations; interpretability
surfaces those failures.

**Q: What's special about histopathology image analysis?**
A: (1) Gigapixel images (100k × 100k pixels) — can't fit in memory; tile-based
processing required. (2) Stain variability — H&E staining varies across labs
and over time; stain normalisation is critical. (3) Weak labels — annotations
are usually slide-level diagnoses, not per-pixel. (4) Strong class imbalance —
most tissue is normal; pathological regions are rare. (5) Multi-scale structure
— relevant features exist from cellular (20×) to tissue-architecture (1×) levels.

## 13. Practical issues

**Q: How do you handle class imbalance?**
A: Six approaches, often combined:
1. **Class weighting** in the loss (1/frequency)
2. **Resampling** — oversample minority, undersample majority
3. **Synthetic samples** — SMOTE for tabular; augmentation for images
4. **Focal loss** — down-weight easy examples
5. **Two-stage training** — train on balanced subset, fine-tune on imbalanced
6. **Threshold tuning** — pick operating point on PR curve, don't rely on
   default 0.5

The right answer depends on imbalance severity: mild (1:10) → class weighting
suffices; extreme (1:1000+) → focal loss + threshold tuning + careful evaluation.

**Q: Your model trains to high training accuracy but validation is poor. What
now?**
A: Overfitting. Diagnostic checklist:
1. Plot training/validation loss curves — confirm overfitting (val plateaus or
   rises while train falls).
2. Add regularisation: weight decay, dropout, more augmentation.
3. Reduce model capacity if model >> dataset.
4. Get more data or use stronger augmentation.
5. Earlier stopping.
6. Check for data leakage — train/val from same patient/source can cause
   artificially low val loss followed by collapse on truly held-out test.

**Q: How do you detect domain shift at deployment?**
A: Monitor: (1) input statistics (pixel intensity distributions, feature
embedding distributions vs training); (2) prediction confidence distributions
(drops in mean confidence often precede accuracy drops); (3) model performance
on a small ongoing-labeled validation stream if available. Tools: KS-test for
distribution drift, embedding-space clustering, OOD detection methods.

**Q: How would you handle small datasets in CV?**
A: Stacked techniques:
1. Strong augmentation (RandAugment, Mixup, CutMix)
2. Transfer learning from a large pretrained model (or self-supervised
   pretraining on unlabelled data from the same domain)
3. Cross-validation instead of single train/val split (more reliable metrics)
4. Simpler models (linear / classical) as baselines — sometimes they beat deep
   nets at this scale
5. Active learning to prioritise which examples to label next
6. Synthetic data (rendering, GANs)

## 14. Debugging & practical scenarios

**Q: Your loss is NaN. What do you check?**
A: In order:
1. Learning rate too high → divergence. Try LR × 0.1.
2. Exploding gradients → gradient clipping.
3. Numerical issues in custom loss (log(0), sqrt(negative), division by zero).
4. Mixed precision overflow → loss scaling, switch fp16→bf16.
5. Bad data (NaN/Inf in input).
6. BatchNorm with batch size 1 (variance is 0).

**Q: Your model overfits on small data but you can't get more data. What's the
priority list?**
A:
1. Augmentation first (highest ROI, no labelling cost)
2. Pretrain on related data (self-supervised on unlabelled images of the same
   modality)
3. Smaller model / strong regularisation (weight decay 0.01-0.1, dropout if
   appropriate)
4. Cross-validation to use every example for both train and val
5. Ensemble of differently-seeded models (~1-2% accuracy lift typically)

**Q: You deploy a model and accuracy is 5% worse than offline benchmarks. What
do you investigate?**
A:
1. **Preprocessing mismatch** — most common. Different image resize /
   interpolation / colour-space / normalisation between training and inference.
2. **Distribution shift** — production data differs from test set (lighting,
   devices, populations).
3. **Label noise in test set** — the offline benchmark was optimistic.
4. **Adversarial drift** — users' behaviour changes in response to the model.
5. **Quantisation / runtime differences** — fp32 training → int8 inference can
   shift accuracy.

Diagnostic: capture production inputs, run through training pipeline, compare.
The mismatch is usually upstream of the model.

## 15. System design / open-ended

**Q: Design a system to detect tumours in CT scans.**
A: Walk through:
1. **Data**: source labelled CTs (radiologist annotations); deal with class
   imbalance (most patients are healthy).
2. **Preprocessing**: HU windowing for relevant tissue, slice-level vs
   volume-level processing, intensity normalisation.
3. **Model**: 3D U-Net for segmentation, or 2D + temporal aggregation if 3D is
   too expensive.
4. **Loss**: combined BCE + Dice for segmentation with imbalance.
5. **Evaluation**: Dice on tumour regions, sensitivity/specificity at patient
   level (you want to catch every tumour even if some healthy patients get
   flagged for review).
6. **Validation**: hold out by patient (not by slice!) to avoid data leakage.
   Stratify by tumour type/size.
7. **Deployment**: human-in-the-loop — model flags, radiologist confirms. Never
   autonomous in medical imaging without regulatory clearance.
8. **Monitoring**: track confidence distribution, flag novel cases for expert
   review, periodic retraining.

**Q: How would you build a CV system for an autonomous lab where robots image
samples?**
A: Different priorities from clinical CV:
- **Throughput** matters (thousands of samples/hour).
- **Robustness** matters (slight focus drift, illumination changes).
- **Closed loop**: model outputs go back to the robot for next-action decisions;
  latency matters.
- Approach: lightweight CNN (MobileNet/EfficientNet-lite), heavy domain-specific
  augmentation (focus, brightness, plate position), aggressive uptime
  monitoring, automatic alerting when model uncertainty spikes.

## 16. Behavioral / scientific discipline

**Q: Tell me about a time a CV project failed and what you learned.**
A: Frame it as: situation → what went wrong → root cause → lesson. Avoid
blaming tools or teammates. *Good answer* structure: "We had impressive offline
metrics but the model collapsed in production. Root cause was data leakage —
train/test came from the same scanning sessions, so the model learned
scanner-specific artifacts, not the disease. The lesson: always split by
*acquisition session* or *patient*, not by image. Now I do that even when it's
inconvenient."

**Q: How do you decide whether to use a classical method vs a deep model?**
A: Three factors:
1. **Data scale**: <1k labelled examples → classical often wins; 10k+ → deep
   usually wins; in between → empirical.
2. **Interpretability requirement**: regulated domains favour classical or
   interpretable deep variants.
3. **Compute budget**: edge deployment without GPU favours classical or
   distilled small models.

Default to classical for the first baseline; switch to deep when you can
quantify the lift.

**Q: How do you handle disagreement with a stakeholder who wants a more complex
model?**
A: Frame as a hypothesis to test, not a debate. "Let's set up an A/B comparison
— current baseline vs the proposed model, on a frozen test set, with predefined
success criteria. If the complex model beats the baseline by Δ% on this metric,
we ship it; otherwise we keep the baseline." Removes the argument from opinion
to data.

## 17. Rapid-fire short answers

**Q: What's the difference between bilinear and bicubic interpolation?**
A: Bilinear uses 4 neighbours, linear in each direction. Bicubic uses 16
neighbours, cubic polynomial — smoother but slower. Bicubic for high-quality
resize; bilinear for speed; nearest-neighbour for label masks (avoids
introducing new label values).

**Q: What's a confusion matrix?**
A: Square table of true class × predicted class. Diagonal = correct predictions.
Reveals which classes the model confuses with which — e.g., "we mostly
misclassify reactive microglia as resting microglia" is more useful than "F1
is 0.7."

**Q: What's the curse of dimensionality in computer vision?**
A: As feature dimensions grow, the data becomes increasingly sparse in feature
space; distances become less meaningful (everything is far from everything
else). Why dimensionality reduction (PCA) matters before linear classifiers on
high-dim features.

**Q: What's Hough transform for?**
A: Detecting parametric shapes (lines, circles) in edge maps. Each edge pixel
votes for all parameter combinations that could pass through it; peaks in the
parameter space = detected shapes. Classical, robust to noise, surprisingly
competitive for simple geometric primitives.

**Q: Define homography.**
A: A 3×3 matrix relating two planar views of the same scene (e.g., two photos
of a flat surface, or any two photos when the camera only rotates). Used in
panorama stitching, planar object tracking, AR registration. Has 8 DOF (one
scale free).

**Q: Difference between epipolar geometry and homography?**
A: Homography: relates two views of a *plane*. Epipolar geometry: relates two
views of an arbitrary 3D scene (point in image 1 → line in image 2). Used in
stereo vision.

**Q: What's optical flow?**
A: Per-pixel motion vectors between two consecutive frames. Classical:
Lucas-Kanade (sparse), Horn-Schunck (dense). Modern: FlowNet, RAFT (deep
learning). Used in video understanding, action recognition, stabilisation.

**Q: What's the difference between SAM and Mask R-CNN?**
A: Mask R-CNN: trained to segment known classes you have labels for. SAM:
prompt-driven, zero-shot — give it a point/box/mask and it returns the
corresponding object's mask, without knowing the object class. Different use
cases — Mask R-CNN for production class-aware segmentation; SAM as a labelling
tool or for novel objects.

## 18. Pharma-specific

**Q: Why is reproducibility a bigger deal in pharma CV than in tech CV?**
A: Pharma is regulated. An FDA submission requires that you can demonstrate
the model produces identical outputs given identical inputs, across versions,
across years, across hardware. Means: deterministic seeds, frozen library
versions, container-based deployment, pickled artifacts checked in. The "move
fast and break things" tech culture doesn't survive contact with this.

**Q: What's a common pitfall in pharma CV from a methodology standpoint?**
A: **Splitting data by image instead of by source.** If you have 100 microscopy
images from 10 patients, splitting train/test by image puts patient images in
both sets. The model overfits to patient-specific artifacts; the test score is
optimistic. Always split by patient/source/acquisition session.

**Q: What does "GxP" mean and why should a CV engineer care?**
A: Good *something* Practice — GMP (Manufacturing), GLP (Laboratory), GCP
(Clinical). Encodes the standards for what's auditable, traceable, and
validateable in pharma operations. CV systems that touch GxP processes need
full audit trails, version control, validation documentation, and change
control. Affects how you design pipelines, not just models.

**Q: Explain the difference between assay development and assay deployment for
CV.**
A:
- **Development**: building the analysis pipeline. Flexibility, prototyping,
  model iteration.
- **Deployment**: the assay runs at scale in production. Locked-down pipeline,
  validated end-to-end, regulatory documentation, monitoring.

The transition between them is where most pharma CV projects get stuck — what
worked for the data scientist on their laptop doesn't survive deployment
without significant engineering for reproducibility and validation.
