## `tune_detection_threshold` Rewritten — Cached-Score Sweep

Old version: for each of 15 thresholds, call `process_image` on each of 3 val
images → **45 full multi-scale detections** per sweep, with only the final
score-filter changing. ~6 h, the reason yesterday's run was stopped.

New version: run multi-scale detection **once per val image** with
`cfg.detection_threshold = -np.inf` to capture every candidate window pre-NMS.
Cache `(all_dets, all_scores, gt_boxes)` per image. The sweep then filters the
cached scores per threshold and runs NMS only on the survivors. **3 detections
total per sweep**, ~30 min instead of 6 h.

### Why filter-then-NMS gives identical above-threshold survivors as NMS-then-filter

NMS is processed in descending score order. For any above-threshold box `B_j`,
its kept/suppressed status depends only on which higher-scoring boxes have
already been kept — none of the lower-scoring (would-be-filtered-out) boxes
matter. So whether we filter before or after NMS, the above-threshold survivors
are the same set. This justifies dropping the per-threshold redetection
entirely, and is recorded in a comment on the sweep loop.

### Why the pyramid loop is inlined, not reusing `multi_scale_detect`

`multi_scale_detect` → `non_max_suppression` returns boxes only — the scores
are dropped at NMS. We need scores to cache, so we can't call it. The clean
alternative would be to refactor `multi_scale_detect` to return `(boxes,
scores)` plus an `apply_nms: bool=True` flag and update the other two callers
(HNM loop + test eval). Decided not to: the inlined block is ~10 lines, the
refactor ripples to two call sites for no near-term gain. Inlined version
stays.

## Why F1 = 0.284 — Quantitative Decomposition

Back-solving the t=3.92 row gives the per-image picture:

| | Per validate image (≈) |
|---|---|
| Sliding-window positions | 45,000 |
| Ground-truth cells | 50 |
| TP | 18 |
| FP | 62 |
| Specificity error | 62 / 44,950 = **0.14%** |

The classifier is actually 99.86% specific — quite good. The 0.284 F1 is not a
classifier-quality failure; it's three structural effects compounding:

1. **Class imbalance at inference.** 0.14% FPR × ~45k negatives still produces
   ~60 FPs/image, against only ~18 TPs. This is the project's own
   "patch metrics ≠ detection quality" thesis quantified — same model gives
   patch F1 ≈ 0.5 on the (balanced) validate split, but detection F1 ≈ 0.28
   when sweeping the full image. ~44% drop, same classifier.
2. **Pyramid coverage gap.** Scales (0.75, 1.0, 1.5) → effective window sizes
   43 / 64 / 85 px. Cells <32 px (p10) and >86 px (p90) fall outside any
   scale's IoU≥0.5 reachability. ~20% of the cell population unreachable. The
   extension to (0.5, 0.75, 1.0, 1.5, 2.0) sketched in [[2026-05-19]] would
   directly recover some of the lost recall, no retraining.
3. **IoU ≥ 0.5 strictness on grid-aligned windows.** No box regression in this
   pipeline (Dalal & Triggs lineage — see [[../research]]). A 32 px cell
   against a 43 px window maxes out at IoU = (32/43)² ≈ 0.55 when perfectly
   centered, dropping below 0.5 within a few pixels of offset. The 8 px stride
   doesn't always land near a cell's centre. A fraction of correctly-classified
   cells fail the IoU match anyway.

### Why this is Beat 3 / Beat 4 of the presentation, not a result-quality problem

`0.5 → 0.28` is the headline punchline replacing the 2023 patch-F1 = 0.92
claim. The drop quantifies why "looks good on paper, fails in production" — a
narrative pharma audiences will recognise. Future fixes (pyramid extension,
higher-capacity model, box regression, more labelled data) are concrete Q&A
follow-ups, not deficiencies to apologise for.

### What the test eval will likely show

Validate F1 is **slightly optimistic** because HNM mined hard-negative crops
from validate-image *backgrounds* into training (no GT contamination, but
specific FP-style examples did leak in). Test images were untouched. Expect
test F1 around or slightly below 0.28.

## Baseline & Augmentation — Framing for Beat 3 Ablation

For Beat 3 ("here's what was wrong + fixed"), the right *internal* baseline is
a **barebones SVM ablated within the current codebase** — same train/val/test
split, same evaluation harness, just turn off everything that was added.
Isolates the engineering contribution from confounders like split choice or
eval bugs.

### Recommended baseline configuration

| Component | Baseline | Current |
|---|---|---|
| Features | HOG-coarse only (Dalal & Triggs) | HOG-fine + HOG-coarse + colour hist + spatial bins + LBP + intensity stats |
| Dim reduction | none | PCA → 4,677 components |
| Hard negatives | random background only | HNM (4 rounds, +13,651 mined) |
| Pyramid | single scale (1.0) | (0.75, 1.0, 1.5) with cross-scale NMS |
| Threshold | default / patch-F1-tuned | detection-F1-tuned at IoU≥0.5 |
| Data leakage | **fixed in both** | fixed |
| Rotation augmentation | **on in both** | on |

Critical: the **leakage fix is applied in both** versions. If the baseline keeps
the leak, the comparison is unfair — the audience can rightly say "you only
beat your old self because you fixed a known bug." The bug fix is *one of the
ablated rows*, not a free pass.

### Why not other baselines

- `SVM_original.ipynb` as-is — has the leakage. Its patch F1 ≈ 0.92 is the
  *misleading headline*, useful for Beat 1's "looked good on paper" claim but
  not as a technical comparison point.
- Random Forest on the same features — changes the classifier, not the
  engineering. Doesn't test the right hypothesis.
- "Just cite the old number" — doesn't show what the new engineering bought,
  because the gap from 0.92 → 0.28 also includes the leakage-fix effect.

### Where rotation augmentation sits in the story — necessity, not improvement

Rotation augmentation is **not** a 2026 addition. It was in the 2023 pipeline
too; what changed is *where it sat in the pipeline order* (i.e. the
leakage fix — augment after split, not before). The augmentation itself is
an architectural necessity, not a claimed engineering win.

**Why it's a necessity for this architecture specifically**: HOG features bin
gradients into orientation channels — rotate the input and you get a *different*
feature vector. LinearSVC has zero rotation invariance baked in. With only
~800 training cells and no architectural invariance, the classifier would
overfit to the orientations that happened to appear in those 11 training
images and miss everything else at inference. Augmentation is how rotation
invariance is delivered **through the data** when it can't come from the model.

**Implication for the ablation table**: rotation augmentation is a *constant
background* in both rows — applied (post-split) in baseline and current alike.
The ablated row adjacent to it is the **data-leakage fix** ("before: augment
then split / after: split then augment"). Conflating "the pipeline needs this
to work" with "the engineering improved over baseline" would be a fair target
for interviewer pushback.

### Backup-slide ablation worth having ready

An optional "with vs without rotation augmentation" comparison (re-train
current pipeline with `augment=False` on the train microglia path → re-evaluate
detection) is strong Q&A material. The recall drop will likely be substantial,
and the framing — *"for a linear classifier on HOG features, augmentation is
the rotation-invariance mechanism the architecture can't provide"* — is the
kind of architectural-trade-off answer pharma audiences will respond to. Not
worth slide-time in the 7 min though; keep it as a backup.
