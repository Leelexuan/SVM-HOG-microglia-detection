### Surprise — Row 4 → Row 5 is slightly **negative** on test

Validate best F1 jumped 0.220 → 0.284 (+29%) but test F1 went 0.220 → 0.211
(−4%). FPs collapse 330 → 128 (−61%) as expected from raising the threshold;
recall halves (0.290 → 0.180), which more than offsets the precision gain.

**Why** — the t=3.92 chosen on validate doesn't transfer cleanly to test. Two
contributing factors, ordered by likely magnitude:

1. **HNM mined on validate.** HNM rounds 1–4 collected false positives from
   validate-image *backgrounds* and added them as training negatives. The
   classifier's score distribution is therefore tighter around validate's
   negatives than around test's. At a high threshold this asymmetry crosses
   into "test FPs that survive HNM's pull also drag TPs down on test."
   [[2026-05-23]] predicted "test F1 around or slightly below 0.28" — the
   direction was right, the magnitude was understated.
2. **Test images have a different score distribution** in general — different
   staining batch, focal plane, etc. The single validate-tuned scalar `t`
   can't compensate.

**Implication for the slide-deck story:** row 5 is the *methodological*
punchline, not a numerical headline. It demonstrates the right way to pick a
threshold — but on this dataset, the validate-tuned threshold doesn't
generalise as cleanly as the patch-tuned `t=1.5` did, because the operating
point sits in a steeper part of the curve. Honest framing for Beat 4:

> *"At a proper detection-tuned operating point on validate we reach F1=0.28.
> On held-out test that drops to F1=0.21 — the FP collapse from threshold
> tuning is real (330 → 128) but the validate-tuned threshold is slightly too
> high for test images, costing recall. With more held-out data or a small
> calibration set per image, that gap closes."*

## Ablation Fairness — Don't Per-Row Tune Row 2 in Isolation

Considered tuning row 2's detection threshold separately to see if its features
"would have looked better" with a row-specific operating point. **Decision: don't.**
Mixing one tuned row with three untuned rows is unfair — it compares a tuned
model against untuned ones and isn't an "+X" ablation step.

Two coherent framings, only one belongs in the 7-min slot:

| | Framing |
|---|---|
| **A (current)** | Fix `t=1.5` across rows 1–4. Row 5 *is* "+ detection-level threshold tuning". Each row's delta isolates that component's effect at the default operating point. Beat 3 punchline lands hard. |
| B | Tune threshold on every row. Shows the "fully realised" F1 each component combo can reach. Row 5 ceases to exist as a row; threshold tuning is folded into every row. Beat 3 dilutes. |

Sticking with A. Side effect — rows 2 and 3 look bad at `t=1.5` partly because
`t=1.5` is the wrong operating point for those configurations. **That is the
point of Beat 3**, not a flaw in the ablation.

## Row 1 vs Row 2 — Are the Added Features Effective? **Yes.**

Pulled both `run_row1.log` and `run_row2.log` and aligned every available
metric. Row 2 wins on every evaluation that doesn't pin a fixed numerical
threshold; loses only where the operating point is fixed at SVM's default
(score > 0) or at the inherited `t=1.5`.

### Full Evidence Table

| Stage | Row 1 (HOG-coarse) | Row 2 (full features) | Δ |
|---|---|---|---|
| **CV F1** best (balanced, 5-fold) | 0.9225 @ C=0.001 | **0.9547 @ C=0.01** | **+0.032** |
| CV F1 — at every C value | 0.888–0.922 | **0.953–0.955** | **+0.03 to +0.07** |
| Std across folds | ±0.0070 | **±0.0045** | more stable |
| Validate patch F1 (imbalanced 153 / 2883) | 0.6004 | **0.7558** | **+0.155** |
| Validate precision | 0.4484 | **0.6229** | **+0.175** |
| Validate recall | 0.9085 | **0.9608** | **+0.052** |
| Test patch recall | 0.9878 | **1.0000** | +0.012 |
| Test patch F1 | **0.4929** | 0.3945 | −0.098 |
| **Detection F1 @ t=1.5** | **0.149** | 0.135 | −0.014 |
| Detection TP | 81 | **91** | **+10** |
| Detection FP | 758 | 1008 | +250 |
| Detection recall | 0.331 | **0.371** | **+0.040** |
| Detection precision | 0.097 | 0.083 | −0.014 |

### Conclusion

Features made the classifier strictly better. Where row 2 loses (test patch F1,
detection F1 at `t=1.5`), it's the operating-point that fails — not the
classifier. Three independent confirmations:

1. **Every C value in row 2's GridSearchCV beats row 1's best**, with lower
   std. Cross-validated on the same train fold — no operating-point confound.
2. **Validate patch F1 jumped 0.60 → 0.76** (+26% relative). Held-out
   imbalanced patches — the closest patch-level proxy for detection conditions.
3. **Detection recall improved (+10 TPs, +0.04 recall)** at the same `t=1.5`.
   Features found cells row 1 missed.

### Why the Operating Point Shifts Between Rows

GridSearchCV chose **C=0.01 for row 2 vs C=0.001 for row 1** — 10× less
regularisation → wider SVM margin → wider score distribution. The number "1.5"
sits at a different percentile of row 2's score distribution than row 1's.
Plotting same threshold on two different P-R curves at different x-coordinates,
not at the same point on the same curve. The +250 FP delta at `t=1.5` is row
2's curve being sampled further left (more permissive), not its curve being
worse.

### Why This Lands as Beat 3, Not a Result-Quality Problem

The row 1 → row 2 F1 dip is itself a **live instance of "patch metrics ≠
detection quality"** — applied to the operating point across feature spaces.
The features lifted the underlying classifier substantially (CV +0.03–0.07,
validate patch +0.155); only the inherited fixed threshold makes the detection
number look worse. Beat 3's "the threshold was the wrong signal" extends
naturally: *"...and the right signal also depends on which feature space you're
operating in."*

### Slide-Ready Phrasing

> *"Adding colour, LBP, and spatial features lifted patch validate F1 from
> 0.60 → 0.76 — a 26% relative improvement. At the inherited threshold the
> detection-level number actually drops, for the same reason the whole
> 'patch ≠ detection' insight exists. Row 5 fixes it."*

## P-R Curve Overlay — Row 1 vs Row 2 on Validate

Done. Script: `2026/pr_curve_row1_vs_row2.py`. Plot:
`2026/pr_curve_row1_vs_row2.png`.

For each row, ran one pre-NMS detection pass per validate image
(scale=1.0 only, matching the row's `cfg.scale_factors`), cached `(box, score,
gt)` per image, then swept ~40 thresholds in `[p5, p99.5]` of the score
distribution, ran NMS + IoU-matched evaluation on the survivors at each
threshold. ~22s for row 1, ~226s for row 2 (full features × 3 scales of LBP).

### Result — Row 2's curve completely dominates Row 1's

|  | Row 1 (HOG-coarse) | Row 2 (full features) |
|---|---|---|
| **Best F1 on validate** | **0.050 @ t=2.27** (P=0.031, R=0.131) | **0.288 @ t=3.81** (P=0.371, R=0.235) |
| @ t=1.5 (the ablation operating point) | P=0.026, R=0.203, F1=0.046 | P=0.069, R=0.359, F1=0.115 |
| Score range swept | 0 → 3.5 | 0 → 3.9 |

Row 2's best F1 is **5.7× row 1's best F1** on the same images at the same
single scale with the same NMS. Row 1's entire P-R curve sits below row 2's
*lowest precision point*. Row 1 cannot be tuned to a working operating point —
it caps out around F1 ≈ 0.05 because every threshold either keeps FPs (low t)
or kills TPs faster than FPs (high t).

### The buried headline — Row 2 best-tuned ≈ Row 4 best-tuned on validate

| | Best F1 on validate | At threshold |
|---|---|---|
| Row 2 (no HNM, no pyramid, just features) | **0.288** | 3.81 |
| Row 4 (+ HNM + multi-scale) | **0.284** | 3.92 |

Once row 2's threshold is properly chosen, **the additional engineering
(multi-scale pyramid + 4 rounds of HNM) buys ~0.004 on validate**. That's a
striking result for the talk: **at a properly tuned operating point on
validate, features alone get you ~95% of the way to the full pipeline.**

Important caveat: this is **validate**. On held-out test the picture changes
because HNM was trained against validate (see Row 5 section above). Row 2 on
test was not threshold-tuned — running its own validate-tuned threshold on
test is the obvious next probe and is `~3 min` to add.

### Why row 1 is so weak on validate (P_max ≈ 0.03) vs row 1 on test (P=0.097)

Validate and test images differ — validate has fewer cells per image (35–66 GT)
vs test (~122 each), so each test image is "denser" and FPs land near real
cells more often (boosting nominal precision at the same FP count). Row 1's
classifier is genuinely weak; what looked OK on test was partly an artifact of
test images being easier than validate at the same operating point.

### What this gives the deck

- **Strong appendix slide / Q&A visual.** "Row 2's curve dominates row 1's at
  every threshold." One image, undeniable.
- **Reinforces Beat 3's threshold thesis.** The deltas between rows at fixed
  `t=1.5` are 5× smaller than the deltas at each row's best-tuned `t`. Where
  you sample on the curve dominates which curve you sample.
- **Reframes Beat 4's "what mattered most".** Honest answer for Q&A: features
  > threshold tuning > HNM > multi-scale, in that order of validate-F1 impact.

## Memory Note for Future Self

When two ablation rows compare at a fixed threshold and the "later" row's F1
is lower, **before concluding the added component failed, check whether the
operating point is the same point on the P-R curve** — i.e., whether the
score distributions are calibrated across rows. A change in `C` (or any
training-time regularisation) shifts the score distribution, so a fixed
numerical threshold samples different curves at different x-coordinates. The
honest check is per-row tuned-threshold F1, or P-R curve overlay.
