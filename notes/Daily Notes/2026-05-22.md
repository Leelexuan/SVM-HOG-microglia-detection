succeeds optimistically; the process is SIGKILL'd when it touches the pages).

## Root Cause — OOM Kill in PCA's Full Dense SVD

Crash is inside `fit_pipeline()` (cell 14), at the PCA step, called from the HNM loop.

**Why PCA blows up:** `cfg.pca_n_components = 0.95` is a *variance fraction* (float).
scikit-learn cannot select components by variance without every singular value, so
it is forced into `svd_solver='full'` — a full dense SVD via `scipy.linalg.svd`.
Confirmed by introspecting the last good `pca.pkl`:

| pca.pkl field | value |
|---|---|
| `_fit_svd_solver` | `full` |
| `n_components_` | 6133 (= 95% variance) |
| `n_features_in_` | 32801 |
| `components_` dtype | float32 |

**The trigger:** HNM Round 1 mined 15,701 hard negatives (it runs at
`detection_threshold=0.0`, collecting *every* SVM-positive window across 3 scales).
`X_train_raw` grew **14,914 → 30,615 rows** (confirmed from the crashed run's
`.npz` headers: `(30615, 32801) float32`).

**The math:** a full SVD of a 30,615 × 32,801 matrix needs, on top of ~12 GB
already live:
- an ~8 GB float64 copy of the input (LAPACK requires float64)
- a LAPACK `gesdd` workspace of ≈ `4·min²` ≈ **30 GB**

→ 3-5× over the 15 GiB ceiling. It ran fine yesterday at 14,914 rows because SVD
workspace scales `O(min²)` — doubling the rows ~quadruples the workspace, off a
cliff that 14,914 sat just under.

## Two Compounding Root Causes

1. **PCA full dense SVD** — `O(min²)` workspace; fundamentally won't scale on a 15 GiB box.
2. **HNM over-mining** — Round 1 added 15,701 negatives (`detection_threshold=0.0`
   collects every SVM-positive window). The training matrix doubles every round;
   rounds 2-4 would OOM even before PCA, just holding the arrays.

Fixing only #1 buys ~1-2 more rounds before it dies again. A run that completes all
4 HNM rounds needs both.

## Variance Curve — Picking n_components

Computed the PCA cumulative-variance curve via the **Gram-matrix trick**
(eigenvalues of `X·Xᵀ`, a 14,914² matrix, equal PCA's explained variances exactly,
at a fraction of the full-SVD memory). Output: `2026/pca_variance_curve.png`.
Validated: the curve crosses 95% at exactly 6133 components, matching `pca.pkl`.

| Variance | Components |
|---|---|
| 80% | 3116 |
| 85% | 3780 |
| **90%** | **4677**  ← chosen |
| 95% | 6133 |
| 99% | 9706 |

No knee — HOG spreads variance thinly across many dimensions. Picked **4677 (90%)**:
small variance loss vs 95%, ~24% fewer dims (faster GridSearchCV), more headroom.

## Fix — Two PCA Solvers Tried

**Attempt 1: `IncrementalPCA` — FAILED.** Chosen because its memory is bounded by
`batch_size` regardless of row count. Verification OOM'd at 13.8 GB RSS. Two missed
factors:
1. It upcasts batches to **float64** internally (`X = X - col_mean`, col_mean is float64).
2. Its inner SVD runs on `(batch_size + n_components)` rows — for n_components=4677
   that is ≥ ~9,400; with batch_size=7000 it was ~11,700 rows → a float64 dense SVD
   needing ~16 GB. The "memory bounded by batch_size" property held, but the bound
   itself was too high.

**Attempt 2: `PCA(svd_solver='randomized')` — WORKS.** Plain `PCA` centers in place
(`X -= mean_`) so it stays **float32**; randomized SVD is `O(n·k)`, not `O(min²)`.
Verified on a simulated 24,000-row matrix: result float32, **peak RSS 10.5 GB** —
~4.5 GB margin under the 15 GiB ceiling.

## Second OOM — randomized PCA Was Not Enough

That relaunched run **also OOM'd** — same spot (`fit_pipeline`'s PCA refit), kernel
killed at 13.6 GB RSS, with `X_train_raw` grown to 27,384 rows (12,470 HNs mined).

The standalone verification (10.5 GB at 24k rows) underestimated because it did not
replicate the *running kernel's* accumulated state: `new_feats` (1.64 GB, never
freed), the cold-start-loaded `pca` object (~0.8 GB), the old `X_train` — plus 27k
vs 24k rows. Real peak ~13.6 GB > ~12 GB usable.

Pattern: two PCA-solver fixes, two OOMs, each landing just over the line. The real
flaw was **architectural** — `fit_pipeline` refits scaler + PCA on the whole growing
raw matrix every round.

## Decisive Fix — Fit scaler + PCA Once

Refactored the HNM loop: fit scaler + PCA **once** on the base 14,914 rows, then per
round only project that round's new crops through the fixed transform and retrain
the SVM. The cumulative raw/scaled matrix is never rebuilt — per-round peak is
bounded by one round's crops (~6 GB). Also more correct: classical HNM retrains the
classifier, not the feature transform. Raw arrays `del`'d after the one-time fit.

## Crop-Tracking Bug Fixed

Found a pre-existing bug: `save_hard_negatives` named crops `hn_{stem}_{i}.jpg`, so
round 2+ **overwrote** round 1's crops, and `new_hn_paths = all_hn_paths[n:]` then
saw no growth → the loop would false-converge after round 1. Never triggered before
because no run finished round 1. Fixed: round-prefixed filenames
`hn_r{round}_{stem}_{i}.jpg`; `save_hard_negatives` now returns the saved paths and
the loop tracks new crops from that directly.

## HNM Diverged — Root Cause + Fix

The run (PID 42144) was stopped mid-round-3: hard-negative counts were **rising**,
not falling — Round 1 mined 11,508, Round 2 mined 15,422 (+34%); per-image
detections up ~40%. HNM is supposed to converge toward zero.

**Root cause:** the loop re-ran `tune_svm` (GridSearchCV) every round, re-picking
`C`. It swung `C` 10.0 → 0.1 between the initial fit and round 1. Each round was
therefore a *brand-new, differently-regularised model*, not an incremental
refinement of one — so there was no stable classifier to converge. `C=0.1` won on
patch-CV-F1 but is a worse *detector* (the project's own "patch metrics ≠ detection
quality" insight) → more false positives → more hard negatives mined → divergence.

**Reference:** Dalal & Triggs (2005), *HOG for Human Detection* (`notes/research.md`).
Their HNM is a two-pass procedure that retrains the **same** SVM with collected
false positives — it does not re-tune the model each pass.

**Fix applied:** `tune_svm` is now called **once**, in the initial fit; `cfg.svm_C`
is then held fixed and every HNM round calls only `train_svm`. Each round refines
the same classifier — restoring the convergence assumption HNM depends on.
`class_weight=None` is kept: with `C` fixed the imbalance no longer destabilises the
fit, and a plain SVM on the bootstrapped set is itself D&T-faithful. Structurally
the loop is now a multi-round, validation-mined variant of D&T (the converging loop
with `break` is a deliberate generalisation of D&T's fixed two passes).

## C-fix Did NOT Work — Real Root Cause: Hard-Negative Colour Swap

The C-fix run (PID 53293) diverged identically: Round 1 11,508 → Round 2 **15,489**
→ Round 3 15,393. Holding `C` fixed changed nothing — **the C-retuning diagnosis
above was wrong.**

A full read-through of the notebook found the actual bug, in `save_hard_negatives`:

- **Base patches** — `extract_patches_from_image` reads with `cv2.imread` (BGR),
  then `save_patch` → `cv2.imwrite` (expects BGR). Correct-colour JPEG.
- **Hard negatives** — the HNM loop reads source images with `mpimg.imread`
  (**RGB**); `save_hard_negatives` crops and calls the *same* `save_patch` →
  `cv2.imwrite` treats the RGB array as BGR → **the saved crop has R/B swapped.**

Verified on `0_s43.png`, identical region: `extract_features` sees a base patch as
R,G,B = [1.8, 6.5, 58.7] but a hard-negative crop of the *same region* as
[58.7, 7.5, 2.7] — red and blue swapped. Every hard negative ever mined was
colour-inverted, so HNM trained the SVM on negatives that do not correspond to the
real false positives → the boundary is distorted, not refined → divergence. This is
also exactly why the C-fix did nothing: the corruption is in the hard-negative
*features*, not in `C`.

**Fix applied** — one line in `save_hard_negatives`, convert the crop RGB→BGR before
saving so its JPEG matches the base-patch convention:
`save_patch(cv2.cvtColor(crop, cv2.COLOR_RGB2BGR), hnm_folder, fname)`. Verified: a
hard-negative crop's features then equal a base patch's for the same region. The
C-fix is kept — tuning `C` once is still reasonable (D&T-faithful) and harmless.

during threshold tuning** (see the efficiency note below).

**HNM converged.** Per-round new hard negatives: **11,508 → 581 → 753 → 809.**
After round 1's 20× collapse the count settles into a low band (~580–810) with a
mild, *decelerating* drift (+172, then +56) — a plateau, not divergence. HNM
converges to a **steady-state residual (~800), not 0**: once the easy hard
negatives are absorbed, each retrain shifts the linear boundary slightly — fixing
the last batch but turning up a different ~800 marginal windows. A linear SVM
cannot drive this to 0; the residual hard negatives genuinely overlap the cell
region in feature space, so lowering it would need a higher-capacity model
(RBF-kernel SVM / CNN). Final training set: **28,565 samples** (14,914 base +

## Inefficiency Found — `tune_detection_threshold`

It calls `process_image` afresh for **every** threshold → re-runs the full
multi-scale sliding window 15 thresholds × 3 val images = **45 times** (~6 hours),
even though the window scores are identical across thresholds and only the final
score-filter changes. This is why the run was stopped at this stage.
