## Bug Fix — Hard Negatives Saved as Black Images

### Root Cause

`mpimg.imread` (matplotlib) loads PNG source images as **float32 in [0.0, 1.0]**.
`cv2.imwrite` expects uint8 [0, 255] for JPEG output. When passed float32, pixel values
like `0.73` truncate to integer `0` → every saved crop is all-black.

**Data flow that exposed the bug:**
```
mpimg.imread(path)           # → float32 [0.0, 1.0]  ← source of truth for sliding window
save_hard_negatives(img, …)  # passes float32 img directly
  crop = image[y0:y1, x0:x1]   # still float32
  save_patch(crop, …)           # cv2.imwrite → black image
```

`sliding_window_detect` already handled this correctly internally:
```python
img_uint8 = img if img.dtype == np.uint8 else (img * 255).astype(np.uint8)
```
but `save_hard_negatives` had no equivalent conversion.

### Fix Applied

In `save_hard_negatives` (cell 27 of `SVM_original_refactored.ipynb`), added two lines
after crop extraction, before the shape check:

```python
crop = image[y0:y1, x0:x1]
if crop.dtype != np.uint8:               # ← added
    crop = (crop * 255).astype(np.uint8) # ← added
if crop.shape[0] > 0 and crop.shape[1] > 0:
    save_patch(crop, hnm_folder, f"hn_{stem}_{i}.jpg")
```

**Action required:** delete existing black crops from `Processed_training_images/HardNegatives/`
and re-run the HNM loop.

## Bug — Inhomogeneous Feature Vectors from HNM Crops

### Root Cause

`multi_scale_detect` maps detections from each pyramid scale back to original image
coordinates using integer division:

```python
mapped = [
    ((int(x0 / scale), int(y0 / scale)), (int(x1 / scale), int(y1 / scale)))
    for (x0, y0), (x1, y1) in dets
]
```

Because `int(x1 / scale) - int(x0 / scale)` ≠ `window_size / scale` exactly (integer
rounding), mapped boxes have **slightly different widths and heights** across positions
(e.g., 85 vs 86 px at scale=0.75, window_size=64).

`save_hard_negatives` crops `image[y0:y1, x0:x1]` using these mapped coordinates.
Crops with different spatial dimensions produce different HOG output lengths when
`extract_features` loads them back. NumPy cannot stack vectors of different lengths →
`ValueError: inhomogeneous shape`.

### Fix (not yet applied — user stopped before write)

Add `window_size` parameter to `save_hard_negatives` and resize every crop before saving:

```python
def save_hard_negatives(
    image, detections, gt_boxes, hnm_folder, stem,
    iou_thresh=0.5,
    window_size=64,        # ← add
):
    ...
    if not is_tp:
        crop = image[y0:y1, x0:x1]
        if crop.dtype != np.uint8:
            crop = (crop * 255).astype(np.uint8)
        if crop.shape[0] > 0 and crop.shape[1] > 0:
            if crop.shape[0] != window_size or crop.shape[1] != window_size:  # ← add
                crop = cv2.resize(crop, (window_size, window_size))           # ← add
            save_patch(crop, hnm_folder, f"hn_{stem}_{i}.jpg")
            saved += 1
```

Call site update:
```python
# old
save_hard_negatives(img, boxes, gt_boxes, cfg.hnm_folder, stem)
# new
save_hard_negatives(img, boxes, gt_boxes, cfg.hnm_folder, stem, window_size=cfg.window_size)
```

## Window Size & Stride Analysis (from Image_ROIs.csv)

### Cell size distribution — 1061 labeled cells

| Percentile | Width (px) | Height (px) |
|---|---|---|
| p10 | 32 | 33 |
| p25 | 40 | 39 |
| median | 52 | 49 |
| p75 | 68 | 62 |
| p90 | 86 | 78 |
| max | 175 | 162 |

### Parameter verdicts

**`window_size = 64` — keep as-is.**
Median cell (52×49px) fits with ~6px context on each side. Changing requires a full
retrain from scratch; not worth it given the median support.

**`cells_per_step = 2` (16px stride) — keep as-is.**
IoU ≥ 0.5 guarantee requires stride ≤ window/2 = 32px. 16px satisfies this at every
scale. Increasing to 4 (32px stride) would be the minimum-safe value for speed, but
current setting is fine.

**`scale_factors = (0.75, 1.0, 1.5)` — needs extending.**

Effective window coverage in original image coords (`window_size / scale`):

| Scale | Effective (px) | Cells covered |
|---|---|---|
| 1.5 | 43 | misses p10 (32px) |
| 1.0 | 64 | median–p75 |
| 0.75 | 85 | just reaches p90 (86px), misses larger |

### Suggested change (not yet applied)

```python
# In Config dataclass — SVM_original_refactored.ipynb, cell 4
scale_factors: tuple = (0.5, 0.75, 1.0, 1.5, 2.0)
```

New effective coverage: 32px → 128px (p10 through ~p97).

**Trade-off:** 5 scales instead of 3 → ~67% more sliding window compute per image.
The scale=2.0 step enlarges the 2048px image to 4096px (4× pixel count at that scale),
so wall-clock impact is asymmetric — the large-scale steps are cheap (shrunk image),
the small-scale steps (1.5, 2.0) are the expensive ones.

## Background Execution on EC2

To keep the notebook running after closing VS Code SSH:

**Option 1 — tmux + papermill (recommended):**
```bash
pip install papermill
tmux new -s hnm
papermill 2026/SVM_original_refactored.ipynb 2026/SVM_original_refactored_output.ipynb
# Detach: Ctrl+B then D
# Reattach: tmux attach -t hnm
```
Preserves cell outputs (metrics, plots) in the output notebook.

**Option 2 — nohup:**
```bash
jupyter nbconvert --to script 2026/SVM_original_refactored.ipynb --output /tmp/nb_run
nohup python /tmp/nb_run.py > /tmp/nb_run.log 2>&1 &
tail -f /tmp/nb_run.log
```
Simpler but loses matplotlib figures.
